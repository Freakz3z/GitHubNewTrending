## 1. dataclaw

> 把你的 Claude Code、Codex、Gemini CLI、OpenCode 和 OpenClaw 对话历史变成结构化数据，一条命令发布到 Hugging Face。

**链接**: https://github.com/peteromallet/dataclaw

**Stars**: 1,626 | **创建时间**: 2026-02-24 | **语言**: Python

### 项目简介

DataClaw 是一个行为艺术项目。Anthropic 用世界上免费共享的信息构建他们的模型，然后引入反乌托邦的数据政策来阻止其他人做同样的事情 —— 拔梯子。DataClaw 让你把梯子扔回去。它生成的数据集是你自己可以分享的。

它解析会话日志，编辑敏感信息和 PII（个人身份信息），将结果上传为即用型数据集到 Hugging Face。

### 核心功能

| 功能 | 说明 |
|------|------|
| 多源支持 | Claude Code, Codex, Gemini CLI, OpenCode, OpenClaw |
| 隐私保护 | 路径匿名化、用户名哈希、密钥检测、邮件编辑 |
| 数据结构化 | 会话、消息、工具调用、Token 使用统计 |
| 分布式数据集 | 每个导出都标记 `dataclaw`，可能形成人类-AI 编程协作的分布式数据集 |

### 技术架构

**隐私与编辑机制**：
1. 路径匿名化 —— 文件路径剥离为项目相对路径
2. 用户名哈希 —— macOS 用户名 + 配置的用户名替换为稳定哈希
3. 密钥检测 —— 正则表达式捕获 JWT 令牌、API 密钥
4. 熵分析 —— 引号中长高熵字符串标记为潜在密钥
5. 邮件编辑 —— 个人邮件地址移除
6. 自定义编辑 —— 额外配置要编辑的字符串和用户名

**数据模式**（每行一个会话）：
```json
{
  "session_id": "abc-123",
  "project": "my-project",
  "model": "claude-opus-4-6",
  "git_branch": "main",
  "start_time": "2025-06-15T10:00:00+00:00",
  "end_time": "2025-06-15T10:30:00+00:00",
  "messages": [...],
  "stats": {...}
}
```

### 工作流程

```
1. 安装: pip install dataclaw
2. 配置源: dataclaw config --source all
3. 列出项目: dataclaw list --source all
4. 配置排除: dataclaw config --exclude "personal-stuff"
5. 本地导出: dataclaw export --no-push
6. 审查确认: dataclaw confirm --full-name "NAME" --attest-...
7. 发布: dataclaw export --publish-attestation "..."
```

### 潜在影响

DataClaw 挑战了 AI 公司的数据霸权。它让用户重新掌握自己与 AI 对话的所有权，将私人协作转化为公共数据集。如果被广泛采用，可能会形成一个庞大的、真正开放的人类-AI 编程协作数据集 —— 比任何公司都更大、更多样化。

## 3. skills-best-practices

> 为 Agent 编写专业级技能、使用 LLM 验证它们，并保持精简的上下文窗口。

**链接**: https://github.com/mgechev/skills-best-practices

**Stars**: 897 | **创建时间**: 2026-02-24

### 项目简介

这份指南解释了如何为 Agent 编写专业级技能，使用 LLM 验证它们，并保持精简的上下文窗口。这是创建 Agent 技能的最佳实践的精简集合。

### 技能结构

每个技能必须遵循这个目录结构：

```
skill-name/
├── SKILL.md              # 必需: 元数据 + 核心指令 (<500 行)
├── scripts/              # 可执行代码 (Python/Bash) 设计为微型 CLI
├── references/           # 补充上下文 (schemas、cheatsheets)
└── assets/               # 输出中使用的模板或静态文件
```

### 核心原则

#### 优化 Frontmatter 以提升可发现性

**name 和 description 字段**是 Agent 在触发技能之前看到的唯一字段。

- **严格命名**: name 字段必须 1-64 个字符，只包含小写字母、数字和连字符（不能连续），并且**必须与父目录名完全匹配**。
- **编写触发优化的描述**: 最多 1,024 个字符。这是 Agent 用于路由的唯一元数据。用第三人称描述能力，并包含"负向触发器"。

#### 渐进式披露和资源管理

- **保持 SKILL.md 精简**: 主文件限制在 **<500 行**。用于导航和主要流程。
- **使用扁平子目录**: 将大量上下文移到标准文件夹。文件严格保持**一级深度**。
- **按需加载 (JiT)**: 显式指示 Agent 何时读取文件。
- **显式路径**: 始终使用带有正斜杠 `/` 的相对路径，无论操作系统。

#### 具体的程序化指令而不是散文

- **使用分步编号**: 定义工作流为严格的时间顺序序列。
- **提供具体模板**: Agent 模式匹配能力出色。
- **用第三人称祈使句写作**: 将指令作为对 Agent 的直接命令。

#### 为重复操作打包确定性脚本

- **卸载脆弱/重复任务**: 如果 Agent 需要解析复杂数据集或查询特定数据库，给它一个经过测试的脚本。
- **优雅处理边缘情况**: 编写返回高度描述性、人类可读错误消息的脚本。

### 验证指南

#### 发现验证

测试 LLM 如何在隔离环境中解释你的描述，防止虚假触发。

#### 逻辑验证

确保分步指令是确定性的，不会迫使 Agent 猜测缺失步骤。

#### 边缘情况测试

强迫 LLM 寻找脆弱性、不支持的配置和 Web 工具固有的失败状态。

#### 架构优化

强制执行渐进式披露模式以减少 token 占用。

### 潜在影响

这是 Agent 生态系统的标准化努力。随着 Agent 技能（如 nanobot 的 skills）数量激增，缺乏标准会导致：
- 可发现性差（技能触发混乱）
- 上下文膨胀（token 浪费）
- 不可靠（指令模糊）

skills-best-practices 提供了明确的架构模式：SKILL.md 作为大脑、scripts 用于确定性任务、references/ 和 assets/ 用于按需加载。它还引入了验证流程 —— 用 LLM 验证 LLM 技能。这暗示 Agent 生态正在走向专业化、工程化。
